# Feature Review v1: Phase 4a Control Plane Core

- Date: 2026-07-21
- Role: `feature-review`
- Owner module: `control-plane`
- Reviewed plan version: `v0.1`
- Verdict: `REVISE`

## Conclusion

The plan has a strong metadata-only Phase 4a boundary, keeps Provider plaintext,
terminal transport, HPKE/Pairwise Roots, Credential Grants, and Desktop key
storage in their correct later phases, and truthfully specifies at-least-once
command delivery rather than distributed exactly-once execution. P0's
documentation/vector reconciliation is directly executable, and the selected
external versions exist with compatible top-level licenses/toolchains.

The feature is not approved because later build phases still require material
security, persistence, and wire-contract choices. P1 also cannot meet its stated
TypeScript generated-client gate with the only pinned TypeScript generator.
The ranked findings below must be resolved in `design.md`, `api.md`, and
`test.md` before implementation starts.

## Ranked findings

### 1. Critical — freeze a real portable-Vault remote-identity envelope and state what bootstrap can prove

Files: `design.md` sections "Device integration", "Bootstrap", and P3;
`api.md` `BootstrapOptionsRequest` and Device storage contracts; `test.md` P2/P3.

The plan says `remote_device_identity` stores Ed25519/X25519 private keys in
portable Vault v1, but the shipped Vault surface only seals Codex credential
JSON in `vault_items`, whose primary/foreign keys and AAD are bound to
CredentialInstance, Account, Device, Provider, and credential revision. The
plan does not freeze a generic remote-key payload, exact AAD/framing, item/key
revision/CAS, DEK wrapping fields, key-rotation transaction, or crash recovery.
An implementer must therefore invent the security boundary that makes the
initial Daemon anchor possible.

`BootstrapOptionsRequest` also has no `storageMode` or durable-key receipt even
though the prose rejects missing portable-Vault metadata and the tests reject a
"Vault-unbacked anchor marker." X25519/Ed25519 PoP proves possession, not that
private keys are encrypted at rest. Freeze the device migration and envelope
contract, define generation/seal-before-options and recovery/rotation behavior,
add the exact request field(s), and describe any Vault-backing value honestly as
a signed client assertion plus tested local invariant rather than a server-
verifiable cryptographic fact.

The bootstrap token is printed once, while restart preserves only its digest;
the plan has no local-only operator flow for a lost token or an expired ceremony.
Freeze a fail-closed reset/rotate command that is available only while there is
no active user/anchor, define its filesystem/console authorization boundary and
audit behavior, and prove it cannot reset an initialized Control Plane.

### 2. High — reconcile the OpenAPI 3.0.3 TypeScript codegen claim and pin the missing runtime/validator

Files: `api.md` "Authority, versions, and generated artifacts"; `design.md` P1;
`test.md` P1; feature brief acceptance criteria.

The brief and acceptance gates require generated Go and TypeScript clients, but
the only TypeScript artifact is `control-plane-v1.ts` generated by
`openapi-typescript 7.13.0`. That package generates types, not a runtime HTTP
client. P6 meanwhile requires pages to use generated protocol clients. The plan
also requires OpenAPI validation without naming/pinning the validator command.
Choose and pin a generated/runtime TypeScript client (including its full lock
and license evidence), or narrow the contract to generated types plus a named,
reviewed handwritten fetch layer. Freeze the exact validator and generation
commands/configuration so P1's drift gate is reproducible without choosing new
tools during the build.

### 3. High — complete the cookie/CSRF, pre-auth mutation, and recovery-code wire contract

Files: `design.md` "Passkeys, recovery, browser sessions, and CSRF"; `api.md`
browser endpoints and `CurrentAuth`; `test.md` P2.

The browser must send a per-session `X-CSRF-Token`, but no response/endpoint
delivers that value, and its rotation behavior is undefined. Conversely,
bootstrap, Passkey login, and recovery verification are pre-session mutations
that cannot satisfy a per-session token, while the test plan says every
mutation tests missing/wrong CSRF. Freeze an endpoint-by-endpoint matrix for
cookie auth, CSRF, exact Origin, Fetch Metadata, and allowed content types;
define CSRF issuance/rotation/expiry without exposing the session cookie.

Recovery is also missing the number of codes, textual encoding/grouping and
normalization rules, per-code salt length/record schema, replacement-code
policy, and exact transition from restricted recovery session to normal
Passkey session. These choices affect entropy, interoperability, UI, database,
and hostile-input tests and cannot be left to P2 implementation.

The API can register a replacement Passkey but cannot list or delete Passkeys
and has no last-usable-credential rule, recent-auth requirement, compromised-
credential response, or recovery-code rotation after recovery. Freeze the
Passkey lifecycle and ensure a deletion/recovery sequence cannot strand the
single user or leave the replaced credential silently active.

### 4. High — define claim/ack/result DTOs and the durable daemon command-reconciliation state machine

Files: `design.md` "Asynchronous Session Commands"; `api.md` Session Command
endpoints; `test.md` P5.

The design introduces a server-generated claim token/digest and attempt number,
but the API defines no claim response or claim/ack/result request bodies that
bind token, attempt, request digest, result, and idempotency. The server state
machine allows an acknowledged command to wait until result/expiry, while a
claim can be requeued, but it does not define reconnect scanning or how a daemon
distinguishes reserved, executing, locally committed, ambiguous, and completed
receipts.

Freeze those DTOs, authorization and stale-attempt rules, server/daemon
transaction ordering, and per-command local atomicity/recovery. In particular,
specify how `start`/`resume` bind the command ID into existing Session
reservation/creation so a crash between Provider/local side effects and receipt
updates cannot create a second Session. Keep the truthful promise as
at-least-once delivery plus a precisely bounded idempotent local effect, not an
unqualified at-most-once assertion.

### 5. High — tombstone GC does not prevent resurrection after revocation or later re-enrollment

Files: `design.md` "Revisioned sync and tombstones"; `api.md` Sync DTOs;
`test.md` P4.

The frozen eligible set stops waiting for a revoked Device and the tombstone is
then deleted after acknowledgements plus 30 days. A revoked/offline Daemon can
retain an old UUID mapping/outbox, later re-enroll under a new Device identity,
and submit the stale resource after the rejecting tombstone has been collected.
The plan has no permanent retired-ID/high-water record or mandatory snapshot
reconciliation that prevents that resurrection.

Define the compact durable deletion watermark/graveyard (or an equivalent
protocol invariant), its retention, and re-enrollment/bootstrap ordering. Also
resolve how `SyncConflict.differences[].base` is produced: the request contains
only `baseRevision` and proposed state, while no revision-history retention is
specified. Freeze whether mixed valid/conflicting changes apply atomically and
exactly what cursor/result advances in that case.

### 6. High — freeze enrollment activation and capability evolution across Phase 4a, 4b, and 5

Files: `api.md` `DeviceCapability`, enrollment, and version contracts;
`design.md` phase boundary and enrollment sections; `test.md` P3/P6.

The closed v1 capability enum contains only Phase 4a metadata/sync/command
values. Phase 4b needs decrypt/realtime/controller capabilities and Phase 5
needs credential-storage/grant capabilities, but v1 rejects unknown enum values
and exposes no capability amendment/re-attestation flow. The initial Daemon
anchor and Phase 4a Web Devices therefore have no specified way to gain later
capabilities without an invented mutation or full re-enrollment. Freeze the
kind/storage-mode capability matrix, delegation subset rules, versioned
extension strategy, and whether later capability elevation requires new
attestation, key revision, or new Device identity. This is necessary to keep
4a's metadata-only claim true without blocking 4b/5.

The enrollment endpoints also do not define which concrete Daemon/Web actor
invokes `request -> prove -> approve -> activate`, how the approving pinned
Device receives and verifies the pending transcript, or the schema and durable
handling of the "bootstrap material needed for device authentication" returned
exactly once by `activate`. The current signed-device-auth flow appears to need
only Device ID/key possession, making that unnamed material ambiguous. Freeze
the CLI/Web application-service flow, all endpoint request/response DTOs,
one-time delivery/retry-loss recovery, and whether activation creates any
additional credential or merely authorizes later challenge exchange.

### 7. Medium — finish UUID mapping ownership for server-created and restored resources

Files: `design.md` "Device integration" and metadata/sync sections; `api.md`
Profile and Sync contracts; `test.md` P3/P4/P5.

The local-to-server mapping rules cover a local row's first projection, but not
the reverse path for a browser-created Profile, a newly enrolled Device's
snapshot, restored backups, or relation mappings that arrive before their
parents. Define who allocates the local opaque ID, the transactional mapping/
projection order, ownership checks for browser-created resources, dependency
ordering, orphan handling, and what happens when a server UUID is already bound
to another local entity/type. P5 command mapping depends on these rules.

## Positive findings

- The X25519 PoP transcript is consistently stated in `design.md` and `api.md`,
  binds purpose/ceremony/subject keys/server ephemeral/challenge/expiry, rejects
  all-zero shared secrets, and keeps the server ephemeral private key
  memory-only. P0 correctly requires new Go/TypeScript vectors before use.
- The restricted typed RFC 8785/JCS attestation direction is appropriate:
  fixed members, both full key digests, sorted/deduplicated capabilities,
  domain-separated length framing, and negative cross-language vectors. P0
  correctly treats the current hex fingerprint and incomplete attestation
  vector as obsolete rather than production evidence.
- UUIDv7 is correctly limited to network/server identities; installed local
  opaque IDs are preserved through durable mappings rather than rewritten.
- The plan truthfully excludes WSS, terminal/Approval plaintext, Pairwise Roots,
  HPKE, Credential Grants, and Desktop key-store implementation from Phase 4a.
- Rollback uses forward migrations plus verified backup/prior binary and does
  not promise destructive down migrations or remote termination of local
  Sessions.

close Findings 1-7 and then resubmit a new plan version for independent review.
No product build phase is authorized by this verdict.
